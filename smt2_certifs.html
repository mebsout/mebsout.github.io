1 Extraction of Certificates in Kind 2

1.1 k-induction

For a k-inductive property P, we just return the certificate \((k, P)\) without figuring out which invariants are necessary for its proof.

1.2 PDR

For PDR, we keep the inductive clauses together in conjunction of the set of properties. These properties P1, …, Pn are added to the frame so they do not appear in the clauses. Certificates are of the form \((1, Ind \land P_{1} \land ... \land P_{n})\).

1.3 Invariants Generation

Each invariant has its own certificate attached to itself. Generally they are generated by k-induction. When they are lifted, the same substitution σ corresponding to the node instance is applied to the certificate (and the invariant) but k remains unchanged: \((k, \phi\sigma)\)

The certificate generated this way for an invariant lifted through a condact is not correct. Note that in this case the certificate will simply fail to be verified. In practice (i.e. on our set of benchmarks) this is not a problem because even the ones with condacts don't require invariants lifted through them.

1.4 Format

Certificates are generated in the form of SMT2 compliant scripts. This file defines several symbols and contains a header which identifies the predicates used to represent the input system and the predicate. 

(set-info :origin "Certificate generated by kind2 v0.1.1-967-g2029544")
(set-info :input "active_standby.kind.lus")
(set-info :status unsat)

(set-info :init  __I__)
(set-info :trans __T__)
(set-info :prop  __P__)

(set-info :certif "(1 , __PHI__)")

Here the symbol __I__ is declared as representing the initial states of the system, the relation transition is denoted by __T__ and __P__ is the original property. The certificate is a pair, here (1, __PHi__) which means k = 1 and __PHI__ stands for the inductive invariant \(\phi\).

We give additional but optional fields sur as the version of Kind 2 which generated the certificate and the input problem. We also state that all the checks that appear in the file should be unsat, so that some solver can raise an error if this is not the case.

The rest of the certificate defines constants, state variables and node symbols as function symbols in FOL. 

;===============================================================================
; INPUT SYSTEM :
;===============================================================================

;-------------------------------------------------------------------------------
; Constants :
;-------------------------------------------------------------------------------
(declare-fun main.__nondet_0 () Bool)
...

;-------------------------------------------------------------------------------
; State variables :
;-------------------------------------------------------------------------------
(declare-fun transSys.x_is_init_x (Int) Bool)
...

;-------------------------------------------------------------------------------
; Function symbols :
;-------------------------------------------------------------------------------
(define-fun __node_init_ActiveStandby ...)
...

The special symbols of the certificate are defined afterwards. 

;-------------------------------------------------------------------------------
; Initial states :
;-------------------------------------------------------------------------------
(define-fun __I__ ((i Int)) Bool ...)

;-------------------------------------------------------------------------------
; Original property :
;-------------------------------------------------------------------------------
(define-fun __P__ ((i Int)) Bool ...)

;-------------------------------------------------------------------------------
; Transition_relation :
;-------------------------------------------------------------------------------
(define-fun __T__ ((i Int) (j Int)) Bool ...)

;-------------------------------------------------------------------------------
; 1-Inductive invariant :
;-------------------------------------------------------------------------------
(define-fun __PHI__ ((i Int)) Bool ...)

Finally the script contains checks to verify the certificate. These checks encode the rules of k-induction and can be checked by any SMTLIB2 compliant solver that support the theories used. 

;===============================================================================
; CERTIFICATE CHECKER :
;===============================================================================

;-------------------------------------------------------------------------------
; Base case :
;-------------------------------------------------------------------------------

(echo "Checking base case")

(push 1)
(assert (and (__I__ 0) (not (__PHI__ 0))))
(check-sat)

(pop 1)


;-------------------------------------------------------------------------------
; 1-Inductiveness :
;-------------------------------------------------------------------------------

(echo "Checking 1-inductive case")

(push 1)
(assert (and (__PHI__ 0) (__T__ 0 1)))
(assert (not (__PHI__ 1)))
(check-sat)
(pop 1)


;-------------------------------------------------------------------------------
; Property subsumption :
;-------------------------------------------------------------------------------

(echo "Checking property subsumption")

(push 1)
(declare-fun __C0 () Int)
(assert (not (=> (__PHI__ __C0) (__P__ __C0))))
(check-sat)
(pop 1)

(exit)

2 Minimization of Certificates

\((k, \phi)\)

Two simplifications are possible. The first one is the k for which \(\phi\) is k-inductive. The bound computed by Kind 2 at runtime might not be the smallest one, so to ensure the certificate is as easy as possible to check we want to find the smallest such k.

2.1 Reducing k: Strategies

For this we only replay the inductive step, following different strategies governed by the flag passed to the option --certif_min:

  • fwd will incrementally try all bounds starting at 1 up to k and stop at the first where we have inductiveness
  • bwd will start from k and go down to 1, stopping at the first one which does not have inductiveness
  • dicho will proceed by dichotomy

An error is returned if we cannot reverify the inductiveness for the bound k given (this is the case for invariants lifted through condacts for example).

2.2 Reducing φ

\(\phi\) can contain invariants that were not usefull to the proof of k-induction, so we want to remove them. Inductive sets of clauses returned by PDR may also contain useless or redundant information.

We identify the part of \(\phi\) which are auxiliary invariants or inductive clauses from the properties themselves.

\[ \phi \equiv \psi_{1} \land ... \land \psi_{n} \land P \]

Then we perform a fixpoint computation to find out which minimal subset R of \(\{ \psi_{1}, ..., \psi_{n} \}\) is inductive with respect to \(P\), i.e. such that

\[ R(0..k-1) \land P(0..k-1) \land T(0..k) ~\models~ R(k) \land P(k) \]

We note \(f(0..k)\) the conjunction \(f(0) \land ... \land f(k)\).

The first check is to ensure that \(P\) can be preserved at k + 1 if we assume the set of invariants to be true in the k-1 previous steps.

\[ \psi_{1}(0..k-1) \land ... \land \psi_{n}(0..k-1) \land P(0..k-1) \land T(0..k) ~\models~ P(k) \]

If this check fails, then \(\phi\) cannot be k-inductive so we can stop right away. On success of this SMT call, we extract the unsat core from which we can identify the elements of \(\{ \psi_{1}, ..., \psi_{/n }\}\) necessary to the proof of preservation of \(P\). Let's call this first subset \(R_{1}\).

Now \(R_{1} \land P\) needs not be k-inductive itself. We verify it with the following check.

\[ R_{1}(0..k-1) \land P(0..k-1) \land T(0..k) ~\models~ R_{1}(k) \land P(k) \]

If this one succeeds then we are happy, we have identified a minimal (but not minimum) subset which is k-inductive together with \(P\). This materializes the useful invariants for the k-inductiveness of \(P\).

If the test fails, \(R_{1} \land P\) is not k-inductive. We need to add back some of the discarded invariants to $R1$. To identify the ones that are necessary, we just loop back to the first test. At worst we are going to add all invariants back.

3 Benchmarks

These runtimes were obtained on Milner by running Kind 2 with the solver Z3 and all engines enabled. The certificates were generated with the options --certif --certif_min fwd. The Z3 binary used was from version 4.3.2. The CVC4 binary used was from version 1.5-prerelease [git master 230dbc7f].

For these benchmarks we report the following:

Kind2 time
the time necessary for Kind 2 to prove (or disprove) all properties
Min.
A checkmark indicates that minimization of the certificate was used
k
the k of the certificate
size
the number of clauses in the k-inductive invariant
min time
the time that was used to minimize the certificate
gen time
the time that was used to generate the certificate (this includes min time)
Z3 check
the time taken by Z3 to check the certificate
CVC4 check
the time taken by CVC4 to check the certificate
