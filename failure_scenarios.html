Where can things go wrong?

This flow chart shows the possible failures in Kind 2 that could remain undetected even with our proof mechanism.

Kind 2 does not parse the model

If the file contains syntax or typing errors, Kind 2 will stop there and exit. No certificates will be produced.

This is the expected result.

Kind 2 fails to detect a syntax/typing error in the model

JKind detects the error

No front end certificate will be generated.

This is the expected result.

JKind fails to detect the error

JKind fails to detect the error but produces a different output

For instance the types of the same variable will be different in JKind and Kind2. In this case, an intermediate observer will be generated but no certificate will be produced or they will buggy and the SMT solver will figure it out.

This is the expected result.

JKind fails to detect the error and produces the same output

If the encoding make that the error not appear in the SMT2 certificate or in the LFSC proof, it is possible for all the certifications and checks to go through

This is a breach of trust (but unlikely).

Kind2 contains a bug in its model checking engines

This could be an error in the method, for instance the version of IC3 that we are using could be unsound in the presence of certain language features or theories. This can also be an implementation error, the method is sound but is implemented incorrectly. In all these cases, Kind 2 could answer that a property is valid when it is not.

Kind 2 crashes

If Kind 2 crashes because of an internal error, no certificate will be produced. We’re specifically interested in the case where Kind 2 crashes during its model checking phase, not during the certificate production phase. In this case, we cannot conclude anything but the absence of certificate tells us that there is nothing to trust.

This is the expected result.

One of the engines crashes

Kind 2 can continue even if one of its engine crashes. Its results will be kept, including the certificates for whichever property or invariant it was able to prove. Kind 2 might however not terminate, for instance if BMC crashes, in this case no certificate is produced. If the certificate are produced and checked they can be trusted so it doesn’t matter that one of the engines stopped.

This is the expected result.

The intermediate certificate is not checked

Most likely, the SMTL-LIB 2 certificate will not be verified by the chosen SMT solver. This means that one or more of the base case, the k-inductive case, or the implication check will fail when verifying the certificate. The case where the solver is buggy is handled in another section.

In this case, the bug in Kind 2 affects the answer and the certificate is able to detect the error after Kind 2 is run.

This is the expected result.

The intermediate certificate and the LFSC proof are checked

Assuming the LFSC checker can be trusted, the LFSC proof of invariance for the original property on the encoded system gives us the assurance that this property is indeed valid. This means that the bug is not affecting the results for this property and so, even though Kind 2 is buggy, we can trust its results for this particular problem.

This is the expected result.

Kind2 contains a bug in its underlying libraries

ZeroMQ (message passing interface) is buggy

The certificates and the proofs cannot be directly affected by this library because they are generated sequentially at the end of the search. However, malfunction of this component can directly affect the results of Kind 2. Suppose a message is lost and not detected as such, and this message sent by the BMC engine contained a counter-example to some property or invariant at bound 3. The other engines will not have this information so the well behavior can be impacted and, more importantly, Kind 2 might fail to return counter-example trace and instead produce a certificate for a false property.

However, we know that this error will be detected in the certificate because there is no k-inductive strengthening for false properties. This error will be detected by the check of the intermediate SMT solver, no proof will be produced.

This is the expected result.

The SMT solver is buggy

The SMT solver crashes

If an instance of an SMT solver crashes, Kind 2 will stop an no certificate will be produced.

This is the expected result.

The SMT solver is incomplete

The SMT solver does not terminate

If the SMT solver does not answer, so will Kind 2. No certificates/proofs will be produced.

This is the expected result.

The SMT solver answers unknown

Kind 2 will terminate and not generate certificates

This is the expected result.

The SMT solver answers sat while the goal is unsat

In the case of BMC, Kind 2 might report a spurious counter-example. This can be problematic if we trust the counter-example
